# Documentation Rules

Update documentation when changing:

- Setup steps
- Environment variables
- Commands
- Public APIs
- CLI behavior
- Configuration
- Deployment
- Database migrations
- Architecture decisions
- Security assumptions
- Operational procedures

## Good documentation

Good docs are:

- Accurate
- Specific
- Current
- Easy to follow
- Written for the next maintainer
- Connected to actual commands

## README

The README should include:

- What the project does
- How to install dependencies
- How to run locally
- How to test
- How to build
- Required environment variables
- Common troubleshooting
- Deployment notes if applicable

## Comments

Use code comments for non-obvious reasoning.

Do not use comments to restate obvious code.

## Decision records

For major decisions, consider adding an ADR or short note with:

- Context
- Decision
- Alternatives considered
- Consequences
